# 要件定義書：SummaryRecorder
## Xperia Ace III 録音＆ハイブリッド要約システム

---

## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| 目的 | 容量・処理能力制限のある端末を「長時間ノンストップ録音・高精度文字起こし・全体要約」の専用デバイスとして運用 |
| ターゲット端末 | Xperia Ace III (RAM 4GB / Snapdragon 480 5G / Android 12〜13) |
| 運用形態 | 個人利用専用（外部配布なし） |
| APIキー管理 | `local.properties` でローカル管理 |

---

## 2. 機能要件

### 2.1 録音機能

| ID | 要件 | 優先度 |
|---|---|---|
| REC-001 | AudioRecord で 16kHz / Mono / 16bit PCM を連続録音する | **Must** |
| REC-002 | 無音ギャップ0秒を実現する（バッファ切替時の音声欠落なし） | **Must** |
| REC-003 | 録音データを約10分（19MB）ごとに自動分割する | **Must** |
| REC-004 | 分割時のヘッダー書き出しはスレッドセーフに行う（Mutex排他制御） | **Must** |
| REC-005 | WAVヘッダーを動的に生成し、`seek(0)` でデータサイズを上書き確定する | **Must** |
| REC-006 | 録音の開始・停止はForeground Service経由で行う | **Must** |

### 2.2 文字起こし機能

| ID | 要件 | 優先度 |
|---|---|---|
| TR-001 | 分割されたWAVファイルを Groq API (Whisper) に送信し文字起こしする | **Must** |
| TR-002 | 音声ファイルは OOM 回避のためストリーム送信（`RequestBody`）する | **Must** |
| TR-003 | 文字起こし結果を Room DB に保存する | **Must** |
| TR-004 | 1ファイル上限 25MB（Groq制限）。19MB（約10分）でクリア | **Must** |

### 2.3 要約機能

| ID | 要件 | 優先度 |
|---|---|---|
| SUM-001 | 全チャンクの文字起こし完了後、Gemini Flash で一括要約する | **Must** |
| SUM-002 | DB内全テキストを結合して Gemini へ送信（6h録音≈15,000トークン、上限の1.5%未満） | **Must** |
| SUM-003 | 要約結果を画面に表示する | **Must** |
| SUM-004 | 要約完了後、該当セッションのチャンクレコードをDBから物理削除する | **Must** |

### 2.4 状態管理・フェールセーフ

| ID | 要件 | 優先度 |
|---|---|---|
| ST-001 | チャンク状態を `[PENDING / UPLOADING / DONE / FAILED]` の4状態で管理 | **Must** |
| ST-002 | Service起動時に `UPLOADING` 状態のレコードを全て `FAILED` にリセットする（自己修復） | **Must** |
| ST-003 | WorkManager で15分間隔・ネットワーク接続時の自動再送を行う | **Must** |
| ST-004 | Room Flow でリアクティブに全完了を監視する | **Must** |

### 2.5 UI

| ID | 要件 | 優先度 |
|---|---|---|
| UI-001 | 録音開始/停止ボタンを表示 | **Must** |
| UI-002 | 各チャンクの状態（PENDING/UPLOADING/DONE/FAILED）をリアルタイム表示 | **Must** |
| UI-003 | 要約結果を表示 | **Must** |
| UI-004 | Jetpack Compose でUI構築 | **Must** |

## 3. 非機能要件

| ID | 要件 | 対策 |
|---|---|---|
| NFR-001 | サーマルスロットリング回避 | 10分/19MBのバッチ通信。発熱モニタリング |
| NFR-002 | Dozeモード回避 | `ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` で手動設定 |
| NFR-003 | OOM回避 | ファイルストリーム送信。メモリに全載せしない |
| NFR-004 | セキュリティ | 個人利用専用かつ外部配布なしのため、APIキー抽出リスクを許容。キー漏洩時の影響は当該APIの利用枠のみ（金融情報等は含まれない） |
| NFR-005 | ストレージ節約 | 文字起こし完了後、音声ファイル即時削除 |
| NFR-006 | バッテリー持続 | 6時間連続録音が可能なバッテリー消費であること。10分バッチ＋Doze回避で達成 |
| NFR-007 | 状態監視方式 | ポーリング禁止。Room Flowによるリアクティブ監視のみ（CPU/バッテリー無駄消費防止） |

---

## 4. 制約事項

| 項目 | 制約 |
|---|---|
| 対象端末 | Xperia Ace III (RAM 4GB) |
| Android バージョン | Android 12 (API 31) 以上 |
| 音声形式 | WAV (16kHz / Mono / 16bit) |
| 1チャンク上限 | 約19MB（約10分） |
| Groq API制限 | 25MB/リクエスト |
| Gemini制限 | 100万トークン（実使用1.5%未満） |
| WorkManager最短間隔 | 15分 |

---

## 5. 用語定義

| 用語 | 意味 |
|---|---|
| チャンク | 10分単位で分割された録音ファイル＋メタデータ |
| SSOT | Single Source of Truth。状態の唯一の真実源（Room DB） |
| ギャップレス | ファイル切替時の無音区間ゼロ |
| ストリーム送信 | ファイルをメモリに全読込せず、直接HTTP bodyとして送信 |
| 自己修復 | クラッシュ後の宙吊り状態（UPLOADING）を自動的にFAILEDへリセット |

## 6. TDD観点：テスト方針

各要件に対するテスト観点を以下に定義する。テストコードは各Phaseで実装前に記述する（Red-Green-Refactor）。

| 要件ID | テスト観点 | テスト種別 |
|---|---|---|
| REC-001〜006 | PCM録音の正常書込、Mutex排他の正当性、19MB閾値での分割、ヘッダー値の妥当性 | 単体 |
| TR-001〜004 | API疎通、ストリーム送信のOOM安全性、モックレスポンスのDB保存 | 単体＋結合 |
| SUM-001〜004 | 全DONE検知のFlow動作、テキスト結合、Gemini呼出、DB物理削除の確認 | 結合 |
| ST-001〜004 | 状態遷移（PENDING→UPLOADING→DONE/FAILED）、自己修復のリセット動作、WorkManager再送 | 単体＋結合 |
| UI-001〜004 | Compose UIの表示・操作（状態反映、ボタン押下） | UI（Robolectric/Compose Test） |
| NFR-003 | 大容量ファイルのストリーム送信時メモリ使用量 | 性能 |
